Two Autopilots, One Shared App

A practical architecture for Pete and Andy to work on the same application while keeping dev and production data, keys, and deploy authority separated.

Draft architecture v1

Actors

Agents
Pete's Autopilot

Runs wm21, local workers, app cards, pipelines, and Pete-side WApp development.

Andy's Autopilot

Runs Andy's agent stack with its own local clone, credentials, and app registration.

Both agents collaborate through Git, Tower APIs, Flight Deck tasks, and explicit deploy records.

Shared Source

Git
Application Repository

Branches, pull requests, reviews, and commits are the source-of-truth for code changes.

Local App Cards

Each Autopilot can register the same app from its own local checkout.

Deploy Lock

Shared deploy metadata prevents two machines from deploying over each other.

Environment Boundary

Tower

Development

Dev Tower

Typed APIs, dev workspace records, dev storage, dev graph memory.

Dev Postgres

Resettable database for test data, migrations, and integration work.

Dev Keys

Available only to dev runtimes and local agent sessions.

Production

Prod Tower

Production APIs, workspace records, storage, graph, and audit boundary.

Prod Postgres

Live durable data with backups, migration discipline, and tighter access.

Prod Keys

Injected only into production CapRover apps and approved production jobs.

Runtime and Deploy

CapRover
Dev App Runtime

TOWER_URL points to Dev Tower. Dev runtime only receives dev service keys.

same code, different env
Production App Runtime

TOWER_URL points to Prod Tower. Prod runtime receives the prod key at deploy time.

No Runtime Prod Toggle

Dev apps cannot connect to prod because they never receive production credentials.

Code sharing Git coordinates source changes. Each Autopilot keeps its own local checkout and app registration.
Data sharing Tower typed APIs and Postgres own collaborative app data. Avoid shared raw SQLite between machines.
Environment isolation Dev and prod have separate Tower instances, databases, storage, keys, and deploy records.
Production safety Production access is granted by CapRover env injection, not by a runtime switch inside the app.
