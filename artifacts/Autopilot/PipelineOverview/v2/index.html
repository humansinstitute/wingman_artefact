Agent Chat Dispatch Paths

Each branch starts from the same hydrated chat thread. The agent or classifier selects one of these outcomes, then deterministic publisher steps post to the correct user-visible surface.

Agent Chat Dispatch Summary 
flowchart TD
  Mention[Chat mention] --> Hydrate[Hydrate thread, channel, scope, refs]
  Hydrate --> Guard{Proceed?}
  Guard -->|no: self or unauthorized| Ignore[No response]
  Guard -->|yes| Classify[Short lookup, classifier, optional agent decision]

  Classify --> Direct[Direct answer]
  Classify --> NeedInput[Needs input]
  Classify --> DocDiscussion[Document discussion child]
  Classify --> TaskBacked[Create or reuse task and start child]
  Classify --> DirectChild[Start direct child pipeline]
  Classify --> Ignore

  Direct --> ChatReply[Reply in same chat thread]
  NeedInput --> ChatQuestion[Ask question in same chat thread]
  DocDiscussion --> DocClose[Document updated or commented]
  DocClose --> ChatSummary[Reply in same chat thread]
  TaskBacked --> TaskProgress[Child reports on task]
  TaskProgress --> ChatStatus[Chat status with task mention]
  DirectChild --> ChildClose[Child closes to chat thread]
  Ignore --> Silent[No user-facing post]
Direct Answer or Short Lookup 
flowchart TD
  Chat[Chat mention] --> Hydrate[Hydrate latest thread]
  Hydrate --> Short[Short lookup or answer-now decision]
  Short --> Draft[Prepare response draft]
  Draft --> Publish[publishFlightDeckResponse]
  Publish --> User[User sees reply in same thread]

Feedback surface

  • Same channel/thread as the triggering message.
  • No task is created.
  • Required data: channelId, threadId, triggering messageId, response draft.
Needs Input 
flowchart TD
  Chat[Chat mention] --> Classify[Classify missing decision or context]
  Classify --> Question[Focused question]
  Question --> Publish[publishFlightDeckResponse]
  Publish --> User[User sees clarification question in chat]

Feedback surface

  • Same chat thread.
  • Used when a concrete repo, target surface, document, or instruction is missing.
  • Should avoid launching child work with guessed context.
Document Discussion Child Pipeline 
flowchart TD
  Chat[Chat mention about a document] --> Route[routeDiscussionChat]
  Route --> Child[document-discussion]
  Child --> Doc[Ensure or update document, add comments if needed]
  Doc --> Publish[publishFlightDeckResponse]
  Publish --> User[User sees chat reply with document status]

Feedback surface

  • Document may be edited or commented by the child pipeline.
  • Chat thread receives the discussion reply or next question.
  • Required data: document mention/id/title when available, discussion goal, source thread.
Task-Backed Child Work 
flowchart TD
  Chat[Chat request for substantial work] --> Plan[Select child pipeline]
  Plan --> Task[Create or reuse in-progress task]
  Task --> Child[startChildPipeline]
  Child --> Software[software-implementation-review-loop]
  Child --> Research[research-and-report]
  Child --> DoReview[do-and-review]
  Software --> TaskClose[Task progress and final review comment]
  Research --> TaskClose
  DoReview --> TaskClose
  TaskClose --> Review[Task moves to review when done]
  Task --> ChatStatus[Chat reply with task mention and launch status]

Feedback surfaces

  • Immediate chat reply confirms task/child launch.
  • Child pipeline reports progress on the Flight Deck task.
  • Task moves to review only after deterministic closeout.
  • Required data: workPlan.taskId, reporting.mode = flightdeck_task, instructions, origin thread.
Direct Child Pipeline Without a Task 
flowchart TD
  Chat[Chat request] --> Direct[start-direct-pipeline]
  Direct --> Child[Selected child pipeline]
  Child --> Closeout{reporting.mode}
  Closeout -->|chat_thread| ChatClose[Final response in original chat thread]
  Closeout -->|missing| OutputOnly[Pipeline output only, no guaranteed user post]

Feedback surface

  • Preferred direct mode uses reporting.mode = chat_thread.
  • If reporting is missing, the run may complete without a user-visible closeout.
  • Required data: source channel/thread/message and clear child pipeline instructions.
Ignore, Self-Authored, or Unauthorized 
flowchart TD
  Chat[Chat event] --> Guard[Policy and actor checks]
  Guard -->|self-authored| Skip[Skip]
  Guard -->|unauthorized actor| Suppress[Suppress]
  Guard -->|agent says ignore| Noop[No response]
  Skip --> Silent[No user-facing post]
  Suppress --> Silent
  Noop --> Silent

Feedback surface

  • No chat reply or task update.
  • Runtime records a suppression/skip diagnostic for observability.

Data Passed Through Chat Dispatch

These are the fields that matter most for selecting a path and returning feedback to the user.

Data Needed for Closeout impact
workspaceId, scopeId, channelId, threadId All chat dispatch paths Lets publishers reply to the same Flight Deck channel/thread.
Triggering message body, sender, attachments, references Classification and agent decision Determines whether the run answers, asks, creates a task, or starts a child.
workPlan.taskId Task-backed child work All child progress and final status return to the task.
workPlan.reporting.mode = flightdeck_task Task-backed closeout Enables task comments and task state transition to review.
workPlan.reporting.mode = chat_thread Direct child closeout Enables final child response in the original chat thread.
workdir and targetSurface Software child work If missing, dispatch should ask for input instead of launching guessed software work.

Other Dispatch Paths

These are shown for context. They remain less expanded because the requested v2 focus is the agent chat dispatch branch.

Task invocation

  • fd-task-invocation loads the invoked task and comments.
  • Agent chooses direct task comment, needs input, ignore, or child work.
  • Child work gets reporting.mode = flightdeck_task.

Document invocation

  • fd-document-invocation loads the document and comments.
  • Agent edits/comments with document tools.
  • Publisher posts a channel summary with a document mention.

Task response

  • fd-agent-dispatch-task-response handles ordinary task assignment/update dispatch.
  • It can launch the same child work pipelines.
  • Feedback returns as task comments and state updates.

Comment response

  • fd-agent-dispatch-comment-response handles tagged task/document comments.
  • Feedback returns to the same task or document comment surface.

Documented v2 Notes

The important rule for chat dispatch is that the return surface should be explicit. Task-backed work needs reporting.mode = flightdeck_task; direct child work needs reporting.mode = chat_thread; missing reporting means the child pipeline may finish without a user-visible closeout.

UI change in v2

  • The agent chat path is now expandable with native collapsible sections.
  • The top-level diagram shows that chat dispatch has multiple internal routes.

Design intent

  • Do not show every step in child pipelines.
  • Show where the branch dispatches and how it closes out to the user.

Operational caveat

  • Code changes still require an Autopilot restart before live dispatch uses newly wired functions.
  • This artifact is documentation only.
